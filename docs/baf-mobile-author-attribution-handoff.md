# BAF handoff: author attribution for mobile receive/unpacking/photo flows

## Контекст
- Проблема проявляється в BAF UI так, що в колонці `Автор` для частини mobile-операцій видно `web`, а не реального оператора.
- Це спостерігається щонайменше в receive / unpacking / photo-related flow.

## Підтверджена причина

### 1. Технічний користувач `web` приходить із Node.js
- `api-nodejs` викликає 1С HTTP API під технічним користувачем `web`.
- Це підтверджується в `api-nodejs/api/config/globalcars.com.ua/configdb.json` для driver `1CHTTP`.

### 2. BAF бере автора із поточної HTTP-сесії
- У BAF для багатьох записів автор проставляється як:
  - `ПараметрыСеанса.ТекущийПользователь`
- Для HTTP-викликів цей користувач і є `web`.

### 3. Legacy mobile app не передає явного `UserIdRef`
- У legacy iOS client для проблемних функцій:
  - `SAVE_PHOTO`
  - `RESIVE_ORDER_BUY`
  - `UNPACKING_ORDER_BUY`
- у payload немає явного `UserIdRef` / `EmployeeIdRef` / `condigoId`.
- Тому без зміни wire contract BAF не може ідеально знати, який саме девайс-користувач виконав дію.

## Цільова модель

### 1. `Автор`
- `Автор` має лишитися стандартним полем BAF.
- У нього треба писати нормального користувача / ПІБ без префіксів.
- Семантику поля `Автор` змінювати не треба.

### 2. `Джерело зміни`
- Треба додати окреме поле / реквізит / колонку `Джерело зміни`.
- Допустимі значення:
  - `1c`
  - `api`
  - `mobile`
  - `system`

### 3. Як це читати
- `Автор` = хто саме зробив зміну
- `Джерело зміни` = через який канал вона була зроблена

## Практичне рішення без зміни mobile app
- Не брати автора напряму з HTTP-сесії `web`.
- Для mobile receive/unpacking/photo scope визначати автора через бізнес-контекст:
  - `ВыкупТовара.Исполнитель`
  - `ЗаказТовара.Исполнитель`
  - документ, який уже посилається на ці сутності
- Якщо визначити користувача не вдається, залишати safe fallback:
  - `ПараметрыСеанса.ТекущийПользователь`
- Для mobile flow окремо проставляти:
  - `Джерело зміни = mobile`

Це не дає device-exact attribution, але прибирає `web` у вже відомому бізнес-flow і дає стабільну атрибуцію через `Исполнитель`, не ламаючи поле `Автор`.

## Пропонований патч у BAF

### 0. Що саме бажано додати в модель
- Окремий реквізит / поле / вимір / ресурс для `Джерело зміни`
- Значення:
  - `1c` — ручна зміна через інтерфейс BAF/1C
  - `api` — інтеграційна зміна через Node.js / зовнішній API без mobile user-context
  - `mobile` — зміна з мобільної апки
  - `system` — автоматична внутрішня зміна через фон / регламент / службову логіку BAF

Мінімально для поточного кейсу достатньо, щоб mobile-related записи отримували:
- коректний `Автор`
- `Джерело зміни = mobile`

### A. Додати helper-функції у `ПолныеПрава`
Файл:
- `baf/baf-configuration/CommonModules/ПолныеПрава/Ext/Module.bsl`

Пропоновані функції:
- `ПолучитьПользователяПоСотруднику(Сотрудник, ПользовательПоУмолчанию = Неопределено)`
- `ПолучитьПользователяПоЗаказуТовара(ЗаказТовара, ПользовательПоУмолчанию = Неопределено)`
- `ПолучитьПользователяПоВыкупуТовара(ВыкупТовара, ПользовательПоУмолчанию = Неопределено)`
- `ПолучитьПользователяПоРаспаковкеЗаказаТовара(РаспаковкаЗаказТовара, ПользовательПоУмолчанию = Неопределено)`
- `ПолучитьПользователяДляДокумента(Документ, ПользовательПоУмолчанию = Неопределено)`

Суть:
- якщо є `Сотрудник`, знайти `Справочник.Пользователи` по `Пользователи.Сотрудник = &Сотрудник`
- якщо є `ЗаказТовара`, взяти `ЗаказТовара.Исполнитель`
- якщо є `ВыкупТовара`, взяти `ВыкупТовара.Исполнитель`
- якщо є `РазпаковкаЗаказТовара`, вийти на її `ЗаказТовара`
- якщо нічого не знайдено, повернути fallback-користувача

Орієнтовний SQL-запит усередині helper-а:

```bsl
Запрос.Текст = "ВЫБРАТЬ ПЕРВЫЕ 1
|	Пользователи.Ссылка КАК Ссылка
|ИЗ
|	Справочник.Пользователи КАК Пользователи
|ГДЕ
|	Пользователи.Сотрудник = &Сотрудник
|	И НЕ Пользователи.ПометкаУдаления";
```

### B. Фото/файли: не ставити автора безумовно як `web`
Файл:
- `baf/baf-configuration/CommonModules/ПолныеПрава/Ext/Module.bsl`

Замість поточного запису автора через `ПараметрыСеанса.ТекущийПользователь`:

- у `ЗаписатьВСписокФайлов(СтруктураФайла)`:
  - проставляти `Менеджер.Автор = ПолучитьПользователяДляДокумента(Менеджер.Документ, ПараметрыСеанса.ТекущийПользователь)`
- у `ЗаписатьВСписокФотографииПоЗаказу(СтруктураФайла)`:
  - проставляти `Менеджер.Автор = ПолучитьПользователяПоЗаказуТовара(Менеджер.Заказ, ПараметрыСеанса.ТекущийПользователь)`

### C. `ФайлыДокументов` record set: не перетирати автора
Файл:
- `baf/baf-configuration/InformationRegisters/ФайлыДокументов/Ext/RecordSetModule.bsl`

Поточна логіка:

```bsl
Запись.Автор = ПараметрыСеанса.ТекущийПользователь;
```

Пропозиція:
- проставляти автора тільки якщо він ще не заданий
- джерело автора: helper `ПолучитьПользователяДляДокумента(...)`

Орієнтовно:

```bsl
Если Не ЗначениеЗаполнено(Запись.Автор) Тогда
	Запись.Автор = ПолныеПрава.ПолучитьПользователяДляДокумента(
		Запись.Документ,
		ПараметрыСеанса.ТекущийПользователь
	);
КонецЕсли;
```

### D. Receive / unpacking flow: підмінити автора на бізнес-виконавця
Файл:
- `baf/baf-configuration/DataProcessors/АРМ_ПрийомРозпаковкаЗамовлень/Forms/Форма/Ext/Form/Module.bsl`

Пропоновані зміни:

1. `ПриходВыкупТовара`
- було:

```bsl
Док.Автор = ПараметрыСеанса.ТекущийПользователь;
```

- має стати:

```bsl
Док.Автор = ПолныеПрава.ПолучитьПользователяПоВыкупуТовара(
	ВыкупТовара,
	ПараметрыСеанса.ТекущийПользователь
);
```

2. `РазпаковкаЗаказТовара`
- було:

```bsl
РазпаковкаЗаказТовара.Автор = ПараметрыСеанса.ТекущийПользователь;
```

- має стати:

```bsl
РазпаковкаЗаказТовара.Автор = ПолныеПрава.ПолучитьПользователяПоЗаказуТовара(
	Товар.ЗаказТовара,
	ПараметрыСеанса.ТекущийПользователь
);
```

3. `ИсходящаяПосылка`
- було:

```bsl
ДокИсходящаяПосылка.Автор = ПараметрыСеанса.ТекущийПользователь;
```

- має стати:

```bsl
ДокИсходящаяПосылка.Автор = ПолныеПрава.ПолучитьПользователяПоЗаказуТовара(
	Товар.ЗаказТовара,
	ПараметрыСеанса.ТекущийПользователь
);
```

## Що це має дати
- У BAF UI в уже підтвердженому mobile scope:
  - в `Автор` має показуватися користувач, який відповідає `Исполнитель` для відповідного викупу / замовлення;
  - у `Джерело зміни` має стояти `mobile`.

Приклад очікуваного відображення:
- `Автор = Іванова Лілія`
- `Джерело зміни = mobile`

## Межі рішення
- Це workaround без перевипуску mobile app.
- Він не дає 100% точного автора для кожного девайса, якщо:
  - операцію виконав не той працівник, який зараз стоїть у `Исполнитель`
  - `Исполнитель` порожній або некоректний
  - у `Справочник.Пользователи` немає зв’язку `Пользователи.Сотрудник`

## Довгостроково правильне рішення
- Додати в mobile / Node / BAF протокол явний `UserIdRef` або `EmployeeIdRef`
- Використовувати його як пріоритетне джерело автора
- `Джерело зміни` виставляти явно з каналу виклику
- Поточний workaround через `Исполнитель` тоді лишити лише як fallback

## План впровадження
- Поточну legacy mobile app не змінюємо.
- Повний правильний fix планується в межах нової Flutter-апки.
- У новому mobile-клієнті треба передавати в mutation-запитах явний `UserIdRef` залогіненого працівника.
- Після цього:
  - Node.js зможе прокидувати реального користувача далі в BAF;
  - BAF зможе визначати не лише канал `mobile`, а й конкретну людину без евристики через `Исполнитель`.
- До моменту переходу на нову апку цей документ описує тимчасовий server-side / BAF-side підхід без зміни legacy mobile app.

## Що ще потенційно лишається проблемним
- У dump ще є інші місця з прямим:
  - `Автор = ПараметрыСеанса.ТекущийПользователь`
- Наприклад:
  - `АРМ_Касса`
  - `АРМ_Склад`
  - окремі form/object modules документів
- Їх варто перевіряти окремо, бо вони не входили в уже підтверджений mobile receive/unpacking/photo scope.

## Статус
- Код у dump **не змінювався як source of truth**.
- Це лише handoff-опис для окремого BAF-розробника.

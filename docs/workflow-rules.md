# Workflow Rules

## Для агентної роботи в цьому workspace

### 1. Починати з контексту
- Якщо задача локальна для одного repo — спочатку читати локальний `AGENTS.md` і `KNOWLEDGE.md`.
- Якщо задача міжрепозиторна — спочатку читати root `AGENTS.md` / `KNOWLEDGE.md`, потім потрібні repo-level docs.

### 2. Межі змін
- Не переносити файли між repo без окремого підтвердження.
- Не робити root-level destructive cleanup.
- У root `logistica/` працювати як у parent repo з submodules, а не як у monorepo.

### 3. Документування
- Крос-проектні правила/контекст — у root docs.
- Локальні технічні рішення — в repo docs.
- Якщо знання важливе для двох частин платформи, допускається короткий дубль у двох knowledge файлах з лінком на первинне джерело.
- Docs-only зміни можна комітити і пушити без окремого підтвердження користувача.
- Але якщо docs належать до тієї ж задачі, що і code change, їх треба по можливості включати в той самий repo-level changeset, а не виносити в зайві follow-up коміти.
- Уникати дрібних серій docs-only комітів, якщо їх можна зібрати в один змістовний апдейт.

### 3.1 Git / submodules
- Зміни в коді спочатку комітити і пушити у відповідний submodule repo.
- Лише після цього комітити в parent `logistica` оновлений gitlink.
- Якщо задача зачіпає кілька child repo одночасно, правильний порядок такий:
  1. довести код і документацію в **кожному** зачепленому child repo;
  2. закомітити зміни в **кожному** child repo;
  3. запушити **кожен** child repo;
  4. тільки після цього зробити **один** parent commit у `logistica`, який фіксує новий узгоджений platform state через submodule pointers.
- Parent repo не повинен комітитися “по дорозі”, якщо ще не закінчені пов’язані docs або code changes в child repo.
- Для міжрепозиторної задачі ціль — не “по одному parent commit на кожен child commit”, а **один логічно завершений changeset**:
  - child repos містять реальні зміни;
  - parent repo лише фіксує, на які саме ревізії child repos тепер посилається workspace.
- Якщо зміна зачіпає кілька repo, документація теж має входити в той самий rollout:
  - root docs — для platform-level правил і крос-репозиторного контексту;
  - child docs — для локальних технічних причин, обмежень і інцидентів.
- Якщо code вже закомічений, але docs ще не встигли доїхати, їх треба дозбирати максимально компактно:
  - один docs commit на repo замість кількох дрібних;
  - після цього одним parent commit оновити pointers.
- Для нового clone:
  - `git clone --recurse-submodules ...`
- Якщо submodules не ініціалізовані:
  - `git submodule update --init --recursive`
- Після зміни `.gitmodules`:
  - `git submodule sync --recursive`

### 4. BAF правила
- `baf/` — це BAF-level hub, не source of truth сам по собі.
- `baf/baf-configuration/` — це dump-каталог усередині BAF repo.
- Не змінювати BAF dump механічно без прямого запиту.

### 5. Rewrite-friendly підхід
- Для legacy-коду пріоритет:
  - зрозуміти фактичний workflow,
  - задокументувати контракт,
  - зробити мінімальний safe-фікс,
  - лише потім планувати більші рефакторинги.
